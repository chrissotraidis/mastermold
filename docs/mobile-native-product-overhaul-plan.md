# Master Mold Mobile-Native Product Overhaul Plan

**Date:** 2026-06-08  
**Source inputs:** user product critique on 2026-06-08, `docs/ref/financial-agent-PRD.md`, `docs/ref/financial-agent-blueprint-v3-buildspec.md`, `spec.md`, current app routes.

## 1. North Star

Master Mold should be a daily financial decision companion, not a dense sample dashboard.

The user opens it each day and gets one clear answer:

> What should I pay attention to today, given my portfolio and the market?

Everything else supports that loop: connect or enter holdings, run market inference, explain the few things that matter, let the user ask follow-up questions anywhere, and record whether Master Mold's calls were useful over time.

The app must be mobile-native first. Desktop can expose more detail, but the primary experience should work comfortably on a phone with short copy, large tap targets, progressive disclosure, and no long pages of explanatory text.

## 2. Product Principles

1. **Daily decision first.** The home screen is a short daily rundown, not a dashboard index.
2. **Portfolio-aware or clearly marked as sample.** Master Mold's advice only feels real when it is based on imported or manually entered holdings.
3. **Few alerts, high trust.** Alerts should answer "why should I care?" in plain language and should be rare enough to matter.
4. **Chat is global.** Master Mold should be available from any page through a persistent head button that opens a compact chat drawer; on phones the head can sit above the bottom nav as a compact launcher, while the top bar also remains a stable entry point.
5. **Short copy, human voice.** Replace AI-ish phrases like "what changed / clear what's noise / tune the rest" with direct financial language.
6. **Explain terms before metrics.** "NVDA volume is 2.1x average" is not useful unless the app says why that matters for the user's position.
7. **No broken buttons.** Any visible action either works, is disabled with a reason, or is removed.
8. **Truthfulness stays, but not as clutter.** Review, track record, and transparency should become one clear "Performance & Trust" area.
9. **Paper trading must have a purpose.** Call it paper trading and explain the paper account, round, position, and outcome simply.
10. **The Brain is the product.** The UI must reflect the long-term system: a daily market reader that remembers sources, scores outcomes, and improves only when evidence is meaningful.

## 3. Target Information Architecture

Replace the current page sprawl with a tighter mobile-first structure.

### Primary Navigation

- **Today**: daily portfolio-aware rundown and 1-3 priority actions.
- **Portfolio**: net worth, holdings, allocation, connected/manual sources.
- **Paper**: renamed from Practice; simulated trades and outcomes.
- **Performance**: combines track record, transparency, model honesty, and review readiness.
- **Settings**: integrations, manual holdings, API keys, preferences.

### Global Surfaces

- **Master Mold Chat Entry**: static Master Mold head button using the corrected inert-socket bitmap. On phones it appears as a compact bottom-right launcher above the bottom nav and remains available in the top bar; on desktop it appears as a larger bottom-right launcher. Opens a chat drawer/sheet over the current page.
- **Alerts Button**: top-right notification icon near profile/settings. Shows a compact alert inbox/drawer, not a full primary tab by default.
- **Profile/Connections Status**: show whether holdings are sample, manually entered, or imported.

## 4. Today Page Redesign

### Current Problems

- Too much text, too many buttons, too many sections.
- The page feels like a report instead of a daily decision companion.
- "Ask about briefing" navigates away and fragments the workflow.
- User cannot tell what Master Mold actually thinks today.
- Buttons duplicate chat instead of making chat ambient.

### Target Experience

The Today page should fit the first useful pass into one phone screen plus one short scroll.

Top section:

- Greeting/status: "Today, I would focus on..."
- Data state: `Sample`, `Manual portfolio`, or `Imported portfolio`.
- One primary action: **Run today's scan** or **Refresh rundown**.
- Last run time and cost if live inference ran.

Daily rundown:

- 1-3 cards maximum.
- Each card uses this structure:
  - **Plain headline:** "NVDA is your main risk today"
  - **Why it matters:** "It is your largest equity holding, and volume/news moved enough to change today's risk."
  - **What to do:** "Review before adding exposure" / "No action" / "Watch earnings"
  - **Confidence and horizon:** short, not buried.
  - Expand for bull case, bear case, sources, and raw metrics.

Portfolio context strip:

- Net worth / sample total.
- Biggest concentration.
- Largest daily mover.
- Cash/stablecoin availability if known.

Remove from first view:

- Repeated "ask about..." buttons.
- Long workflow link grids.
- Deep metrics that belong in Performance.
- Copy that explains the app instead of the decision.

Acceptance:

- At 375px width, no Today section should require horizontal scrolling.
- First viewport shows the daily rundown CTA and at least one recommendation or "nothing urgent" state.
- All chat actions open the global chat drawer instead of navigating away.

## 5. Global Chat Overhaul

### Current Problems

- Chat is isolated as its own page.
- Other pages use many links that send the user away to chat.
- Chat copy and prompt chips feel generic.

### Target Experience

Create a global Master Mold chat entry that stays available without covering core mobile content.

Behavior:

- Static Master Mold head button, no artificial eye overlays.
- On mobile, keep a compact bottom-right Master Mold head above the bottom nav, with the top-bar head as a second entry point.
- On desktop, keep the larger bottom-right launcher for fast access.
- Tap opens a drawer/sheet with chat.
- Chat receives page context automatically:
  - Today: current rundown and portfolio context.
  - Portfolio: selected holding/allocation.
  - Alert drawer: selected alert.
  - Paper: selected simulated trade.
  - Performance: selected past call.
- The drawer can expand to full-screen on mobile.
- Keep `/chat` as a dedicated full-screen route for deep conversations, but it is no longer the main way to ask follow-ups.

Implementation notes:

- Extract `ChatWorkspace` into reusable drawer-friendly components.
- Add a `GlobalAssistant` client component inside `AppShell`.
- Replace "Ask why it matters" links/buttons with drawer open actions carrying structured context.
- Preserve advisory-only language, but move it into a compact footer/disclaimer.

Acceptance:

- Chat works from Today, Portfolio, Paper, Performance, and Settings without leaving the page.
- A prompt launched from an alert includes that alert's plain-language rationale.
- Mobile drawer has large input, clear close, and no clipped controls.

## 6. Alerts Redesign

### Current Problems

- Alert copy is unclear and too AI-ish.
- Alerts are not specific enough about why the user should care.
- "Ask why it matters" does not work.
- "Log decision" does not visibly do anything.
- Acknowledge works but only grays out the item, which is not enough feedback.
- Alerts feel like noisy raw signals instead of trusted interruptions.

### Target Experience

Alerts become a top-right inbox/drawer with only meaningful items.

Alert card structure:

- **Plain title:** "NVDA is moving unusually today"
- **Relevance:** "You hold 28% of equities in NVDA" or "Watchlist only"
- **Reason:** one sentence translating the signal into a decision concern.
- **Suggested response:** "Review before adding", "Ignore unless it breaks $X", "No action needed"
- **Severity:** Urgent / Worth checking / FYI.
- **Raw data hidden by default:** z-score, volume multiple, funding rate, source.

Required actions:

- **Ask Master Mold**: opens chat drawer with alert context.
- **Save as decision**: opens a lightweight decision sheet with fields prefilled.
- **Dismiss**: removes from active inbox and shows undo.
- **Useful / Not useful**: short feedback control, not noisy copy.

Copy examples:

- Replace "What changed: 4 flagged. Clear what's noise. Tell me what was useful so I tune the rest."
- With: "4 items moved enough to check. Dismiss the noise; I will use that feedback next time."

Acceptance:

- Every alert explains why it matters to the portfolio or says "watchlist only."
- No alert action is a dead button.
- Acknowledged/dismissed alerts move to a secondary "Dismissed" section with undo.
- Alert drawer is usable on mobile and does not require visiting a separate alerts page.

## 7. Portfolio Redesign

### Current Problems

- Overview charts are not useful.
- Asset price mini charts on the overview feel cluttered and low-value.
- Copy feels AI-generated.
- The page does not feel enough like a personal finance app.

### Target Experience

Portfolio should feel closer to Robinhood or a clean net-worth tracker.

Top overview:

- Total value / net worth.
- Daily change and percentage.
- Cash / buying power if available.
- Data source status: imported, manual, or sample.

Charts:

- Replace overview asset price bars with a simple net-worth-over-time chart.
- Add allocation chart: equities / crypto / DeFi / cash.
- Move per-asset price charts to holding detail screens or expandable rows.

Holdings:

- Mobile-friendly list first, table only on desktop.
- Each holding shows symbol, name, value, daily change, and portfolio share.
- Tap a holding to see concentration, thesis, recent changes, and related alerts.

Manual data:

- Add or improve manual holdings entry so the product can be useful before integrations are live.
- Manual holdings must feed Today, alerts relevance, and chat context.

Acceptance:

- Portfolio first screen shows total value, change, and source status.
- Net-worth-over-time replaces current overview asset bar charts.
- Mobile holdings list is readable without sideways scrolling.

## 8. Paper Trading Redesign

### Current Problems

- "Practice" is unclear.
- The purpose of the section is hard to understand.
- Copy feels abstract and AI-ish.
- User cannot tell whether this is paper trading, prediction scoring, or training Master Mold.

### Rename

Rename **Practice** to **Paper** or **Paper Trading**.

### Target Experience

This area should answer:

> If this idea were tested as a paper trade, what would have happened?

Core concepts:

- Paper account with simulator dollars.
- Simulated positions.
- Open ideas.
- Closed ideas and result.
- Master Mold's call vs user's call, where relevant.

Flow:

1. Choose an asset or accept a suggestion from Today.
2. Pick direction, paper size, confidence, and time horizon.
3. Submit a simulated trade.
4. Later, score whether the idea was right and what the paper result would be.

Copy:

- Replace abstract "arena" / "calibration" wording with "paper trade", "result", "what happened", "what we learned."
- Keep scoring, but make it secondary.

Acceptance:

- User understands within 10 seconds that no real money is used.
- A visible paper account exists.
- Submitting a paper trade visibly changes the page.
- Paper trades can be created from Today and Alerts.

## 9. Performance & Trust Redesign

### Current Problems

- Track Record feels useless as a standalone page.
- Transparency is a dense, confusing disclosure wall.
- Review readiness is important but not understandable.
- The user does not care about raw truthfulness text unless it helps calibrate trust.

### Target Experience

Combine Track Record and Transparency into **Performance**.

Sections:

- **How Master Mold has done:** calls made, closed calls, hit rate, average result, confidence calibration.
- **What it learned:** strategy beliefs and updates, clearly marked as internal learning.
- **What is real vs sample:** compact truthfulness summary.
- **Limitations:** what Master Mold cannot do yet.
- **Evidence log:** reviewer/dev details behind a collapsible advanced section.

Copy:

- Avoid legalistic walls of text.
- Prefer simple truth:
  - "Live: chat inference with your key."
  - "Sample: portfolio values unless you connect or enter holdings."
  - "Not built: always-on market brain."

Acceptance:

- `/review` may remain for RDS/reviewer compatibility, but the user-facing nav should point to Performance.
- Track record and transparency no longer compete as separate concepts.
- The page has a short summary first and advanced details later.

## 10. Settings & Integrations

### Current Problems

- Settings direction is good, but integrations must be real, researched, and honest.
- Connections should not imply they work until verified.

### Target Experience

Settings should separate:

- **Connect accounts**: Robinhood via SnapTrade, Coinbase, on-chain wallets.
- **Manual portfolio**: add/edit holdings without external accounts.
- **Live chat keys**: OpenRouter/OpenAI/Anthropic, with test buttons.
- **Preferences**: risk posture, asset focus, alert sensitivity.
- **Data & privacy**: export, delete local profile, explain what leaves the device/server.

Implementation research required before enabling connection claims:

- Verify current official SnapTrade Robinhood read-only flow and free-tier constraints.
- Verify current Coinbase read/view API path and scopes.
- Verify current Zerion or replacement wallet portfolio API.
- Verify where keys are stored and whether local browser storage is acceptable for the current app.
- Add connection tests that show success/failure clearly.

Acceptance:

- No integration says "connected" unless it has a working test.
- Every integration shows exact permission scope in plain English.
- Manual holdings work even with zero external connections.

## 11. Brain & Inference Roadmap

### Current Gap

The current app does not yet maintain the real Brain described in the PRD. It has seeded sample data, some live inference paths, and engine artifacts, but it does not continuously scan, remember, and learn the market.

### Required Product Direction

The Brain must become a durable subsystem that:

- Initializes on first setup.
- Builds a market snapshot from stocks, crypto, news, funding, and on-chain data.
- Maintains memory with event time and knowledge time.
- Reads the user's connected or manual portfolio.
- Produces a daily portfolio-aware rundown.
- Logs every suggestion before outcome.
- Scores outcomes after the stated horizon.
- Updates internal beliefs only when evidence is meaningful.

Milestones:

1. **Manual portfolio-aware inference:** use user-entered holdings as context.
2. **Connected portfolio-aware inference:** use read-only integrations.
3. **Market scan ingestion:** scheduled market/news/crypto scan.
4. **Persistent memory:** store facts, embeddings, decisions, outcomes.
5. **Outcome learning:** reflection updates and confidence calibration.
6. **Always-on daily run:** scheduled scan plus push-ready alert inbox.

Acceptance:

- Today rundown changes when portfolio holdings change.
- Master Mold can explain what sources it used.
- Suggestions are saved before outcomes.
- The system can say "nothing urgent" without manufacturing advice.

## 13. Copy Rewrite Rules

Use plain financial language. Avoid sounding like an AI prototype.

Rules:

- Say "why it matters" in terms of the user's money, concentration, risk, or decision.
- Avoid "interrogate", "surface", "tune", "agentic", "signal hygiene", "clear noise" in user-facing copy.
- Prefer:
  - "Check this"
  - "No action"
  - "Worth watching"
  - "This affects the largest visible holding"
  - "Paper trade"
  - "What happened"
  - "What I got wrong"
- Hide raw metrics until the user expands.
- Every technical metric must have a translation.

Example:

- Bad: "NVDA volume 2.1x average."
- Better: "NVDA is trading much more than usual. Because it is your largest equity holding, this can change your portfolio risk today."

## 16. Immediate Next Build Target

Start with **Phase 7: Brain, import, and evaluation foundation**.

Reason: the mobile shell, global chat, alert drawer, Today rewrite, Portfolio/Paper/Performance cleanup, connection checks, manual holdings, and truthfulness surfaces now have local evidence. Repeating Phase 1-6 cleanup would be churn. The next meaningful product step is to make the Brain and portfolio data path real without overstating what is live.

Next implementation batch:

1. **Scheduled market scan:** add a local scheduler/service boundary that can run the saved-scan pipeline on a cadence, record run status, and fail visibly without changing recommendations from stale data.
2. **Portfolio import hardening:** the explicit import path exists; next harden service coverage, pricing gaps, refresh cadence, and source evidence before treating it as complete connected-account aggregation.
3. **Brain memory storage:** persist market facts, portfolio facts, decisions, and outcomes with event time and knowledge time, then expose exactly what sources a Today recommendation used.
4. **Daily rundown generation:** generate the Today priorities from Brain memory plus the visible portfolio, with a designed `nothing urgent` state when nothing clears the bar.
5. **Evaluation harness slice:** add a small honest scoring surface for forward-only calls before attempting DSR/PBO/MinTRL or Alpaca paper live-shadow.
6. **Safety gates stay enforced:** keep read-only endpoint checks, no execution routes, and preview-only Executor copy green while adding the above.

Do not mark the full PRD complete until scheduled broad scanning, daily connected-portfolio refresh/coverage, forward evaluation, and the live Executor boundary have their own current evidence.

- **Mobile shell:** Today, Alerts, briefing detail, Portfolio, Paper, Performance/Trust, Journal, and Settings render at phone width without horizontal overflow in local browser checks.

- **Paper trading:** The visible product language is now Paper/Paper trading, paper account, paper trades, result, and what happened. No real money movement is implied.
- **Performance/Trust:** Track record, disclosure, model honesty, screener feedback, and review readiness are consolidated under Performance/Trust while `/review` remains reachable for RDS.
- **First-run welcome:** On a 375px phone viewport, the welcome screen now shows the app name, advisory truth, Explore sample data, Set up profile, and the name field without a long first scroll.
- **Inference:** OpenRouter model testing passes locally when the server has the key. Chat uses live inference when available and remains advisory only.

- **Replay controls:** Portfolio, Journal, Paper, and Executor now expose the point-in-time Rewind control where the data layer supports as-of replay. Review copy no longer claims every page can time-travel.
- **Alert clarity:** The full Alerts page now matches the alert drawer: "Why it matters" opens a plain-language explanation first, with raw screener detail secondary.

- **Walkthrough checklist cleanup:** The RDS local review helper is behind a collapsed "Local walkthrough checklist" section on Performance, and its visible copy reads like a local walkthrough rather than an evidence console.

### Still Roadmap / Not Done

- **Always-on Brain ingestion:** The app has on-demand market-memory initialization, durable memory scaffolding, and a local schedule-check endpoint, but broad scheduled internet/news/market scanning is not automatic yet.
- **Read-only account import:** Explicit provider import is wired, but broad connected-account coverage, pricing for unknown assets, and scheduled refresh are not complete yet.
- **Full eval harness:** The UI shows local product metrics and outcome tracking, but DSR/PBO/MinTRL, Alpaca paper live-shadow, and post-cutoff validation are not live.
- **Live Web3 executor:** The monitor and kill-switch flow are display/local-state only. No custody, chain calls, signing, Tenderly, or Blockaid path is live in this app.
